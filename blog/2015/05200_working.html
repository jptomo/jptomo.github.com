仕事の進め方雑感 2015春¶

2015/05/20

仕事の進め方最近感じたことなどまとめてみる。

  • 一人でのミーティング
  • 課題のハンドリング
  • 修正しすぎ問題
  • 立ち位置

一人でのミーティング¶

平たく言うと、一人でミーティングすると辛いなと思う。

通常はプロジェクト PM にミーティングをファシリテートしてもらい、 必要な時に僕などエンジニアが各議論に返答をする進め方をしている。

ただ、この前ややこみ入った問題があって、ただシングルイシューのミーティングだったので 一人で進められるかなと、お客さんに来社いただきミーティングをした。

ミーティングを進めていくなかで、舵取りと個別の返答を 両方しなくてはならなくなり、ただ、それぞれの役割にやはり差があって、難しい。

最終的にお客さんに舵取りを任せてしまい、それはそれでよいと思うけど、力不足。

課題のハンドリング¶

仕様がふわふわしている課題って、対応が大変だよねってことで。

最初中規模の機能開発があって、対応していた。

途中で分割してリリースしましょうとなった。

しかし、開発終わったと思ったら分割して後からで良いよとなっていた箇所が 実は分割できないとわかった。

実は、お客さん側でシステムのマニュアルを作成してもらっていて、 同時でないとその修正が辛かったりする感じ。

他にも、お客さんが作業をアウトソーシングしていて、 アウトソーシング先に修正箇所の伝達、というか教育が必要で、同時リリースでないと辛いと。

後は、お客さんの中で、社内SE部門があり、その部門で弊社との機能開発の進捗を管理しているのだが、 今回の機能がシステム横断的な機能であって、お客さんのなかで仕様がまとまってないぶぶんもあって、 そういうのは、大変だけどしょうがないよね。

最終的にお客さんで利用している別の機能の開発が間に合わずリリースが伸びた。

社内的には、お前きちんと機能開発最後までできるのかよと不安をばらまいてしまい、 申し訳なかった。

なんか、多分、どこかの段階で警戒して、問題にならないようにするべきだったんだろうなあと。

最終的にどのように落ち着くべきか、ハンドリングできないとね。

難しい。

修正しすぎ問題¶

口をすっぱくして言われてしまっている。

機能開発やバグ修正で直接は関係のない場所を修正してしまう件。

3年程度継続で開発されているシステムで、開発者が入れ替わりがある程度あり、 うまくかけてなさそうな箇所をついでに直してしまう場合がある。

しかし、レビュー時に何故となり大変だし、修正時に typo などでバグを埋め込み品質が悪くなることも 気になる。ついでに修正した場合、お客さんに説明できないでしょと。

対応する場合は別の枠、チケットでやりましょうと。

システムがより良い状態であれと思い、ショートカットしてやってしまいることは 自覚しているので、プロジェクトで受け入れ易い形で修正をしていけたら なあと思う。

立ち位置¶

将来こんなことやりたいな、このような立ち位置でいたいなと見えておらず、 何とか見えるように進めていきたいとは思うけど。

個人的には MySQL / PostgreSQL の RDBMS まわりの差を理解できると良いなと。

memcached / redis や solr などあるが、 web で最終的に重要なのは DB ではないかなって思って探っている感じ。

もう歳だし、PG定年に近くなっている感じ。
